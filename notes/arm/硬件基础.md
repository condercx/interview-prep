#### NAND FLASH 和NOR FLASH异同？

##### 📊 对比表格

| 类别 | NOR Flash | NAND Flash |
|------|-----------|------------|
| **读取方式** | ✅ **随机访问**：像SRAM，可直接指针读取<br>`uint16_t *p = (uint16_t*)0x08000000; val = *p;` | 🔁 **顺序访问**：需通过命令序列<br>`发送读命令→发地址→查状态→读数据寄存器` |
| **读取速度** | ⚡ 快（随机读取~100ns） | 📦 快（页读取~25μs），但**不支持字节级随机读** |
| **写入速度** | 🐌 慢（~10μs/字节），需先擦除 | ⚡ 快（~200μs/页），页编程效率高 |
| **擦除速度** | 🐌 较慢（50ms~500ms/扇区） | ⚡ 快（2ms~5ms/块） |
| **XIP支持** | ✅ **支持**（代码可直接在Flash上执行） | ❌ **不支持**（需拷贝到RAM执行） |
| **可靠性** | ✅ 高，比特错误率<10⁻¹²，通常无需ECC | ⚠️ 较低，比特翻转常见，**必须配备ECC+坏块管理** |
| **接口类型** | 📡 **并行总线**：地址/数据分离，类SRAM接口 | 🔌 **I/O复用接口**：8/16位数据总线，命令/地址/数据复用 |
| **擦写寿命** | 🔁 **1万~10万次**（块级） | 🔁 **SLC:10万次 / MLC:3千次 / TLC:1千次 / QLC:500次** |
| **数据保持** | 🗓️ 10 ~ 20年（-40℃ ~ 85℃） | 🗓️ 1~10年（随P/E次数和温度衰减） |
| **主流容量** | 📦 128KB ~ 2GB | 📦 1GB ~ **32TB+**（3D NAND） |
| **单位成本** | 💰 高（$0.5~5/MB） | 💰 低（$0.01~0.1/MB） |
| **主要用途** | 🔧 代码存储、Bootloader、配置参数、工业控制 | 💾 大容量存储：SSD、手机存储、U盘、视频数据 |

##### 注意：
在典型的嵌入式SoC架构（如S3C2440、STM32等）中，NOR Flash和NAND Flash的"0地址"确实不冲突，因为它们访问机制完全不同。NOR Flash占用BANK地址空间，NAND Flash不占用BANK地址，0地址是内部的。

---

##### 🔧 相同点

```
1️⃣ 非易失性存储：断电数据不丢失
2️⃣ 写入前必须擦除：写操作只能 1→0，擦除才能 0→1
3️⃣ 擦除单元：均以"块(Block)"为最小擦除单位
4️⃣ 存储原理：基于浮栅晶体管(Floating Gate)或电荷俘获(CTF)技术
5️⃣ 接口演进：均发展出SPI等串行接口以降低引脚数（SPI NOR / SPI NAND）
```

---

##### 🎯 选型建议

```
✅ 选 NOR Flash 当：
   • 需要XIP执行代码（如Bootloader、固件）
   • 容量需求 < 256MB
   • 对可靠性/实时性要求极高（汽车、工业）
   • 系统需要快速启动

✅ 选 NAND Flash 当：
   • 需要大容量数据存储（>512MB）
   • 成本敏感，追求$/GB最优
   • 数据顺序读写为主（文件、多媒体）
   • 可接受软件层管理（FTL、ECC、坏块处理）

🔄 混合架构（常见）：
   NOR(启动代码) + NAND(主存储) → 兼顾启动速度与存储容量
```

#### CPU,MPU,MCU,SOC,SOPC联系与差别？

先说联系：

它们是从“计算核心”到“系统集成度”再到“可编程硬件”的逐步演进关系：

`CPU -> MPU -> MCU/SoC -> SoPC`

1. **CPU（Central Processing Unit）**

   计算机的运算和控制核心，本质是执行指令的核心部件。
   典型工作流程：取指（Fetch）-> 译码（Decode）-> 执行（Execute）-> 写回（Writeback）。

2. **MPU（Micro Processor Unit）微处理器**

   可理解为“更完整/更强的 CPU 芯片”，但通常**不内置**可直接运行程序所需的 RAM/ROM。
   因此一般需要外挂存储器和外设后，系统才能运行。

3. **MCU（Micro Controller Unit）微控制器**

   在一颗芯片内集成了 `CPU + RAM + ROM/Flash + 定时器 + 常用I/O外设`。
   集成度高，通常加少量外围器件就能直接跑程序，常见于单片机场景（如 51/AVR/STM32）。

4. **SoC（System on Chip）片上系统**

   系统级单芯片，集成度和复杂度通常高于 MCU，既有较强处理能力，也有较丰富片上资源。
   常用于可运行操作系统的场景（如 Linux/Android 嵌入式平台）。

5. **SoPC（System on a Programmable Chip）可编程片上系统**

   基于可编程逻辑（典型如 FPGA）的系统级方案。
   与前面“硬件结构基本固化”的芯片不同，SoPC 的**硬件结构可按需求配置**，软件也可配置。

---

##### 核心差别（面试高频）

1. **是否内置存储与外设**

   MPU 通常不内置完整 RAM/ROM 与外设，MCU/SoC 通常内置更多。

2. **能否“最小系统直接运行代码”**

   MCU 往往可以；MPU 通常需要外扩 RAM/ROM 后才能跑完整系统。

3. **系统复杂度与应用层级**

   MCU 偏控制类实时任务；SoC 偏复杂系统任务（多媒体、网络、操作系统）。

4. **硬件可重构能力**

   SoPC 最大特点是硬件可编程、可重构；CPU/MPU/MCU/SoC 的硬件结构通常在出厂时已基本固定。

---

##### 一句话记忆

`CPU` 是“核”；`MPU` 是“强核但要外扩”；`MCU` 是“单片小系统”；`SoC` 是“单片大系统”；`SoPC` 是“可重构系统芯片”。

#### 什么是交叉编译？

交叉编译（Cross Compile）指的是：

> 在一种平台上生成可在另一种平台运行的可执行程序。

通常是：

- **编译主机（Host）**：x86 PC（Windows/Linux）
- **目标平台（Target）**：ARM/MIPS/RISC-V 开发板

也就是说，代码在 PC 上编译，但产物是在板子上运行。

#### 为什么嵌入式里必须交叉编译？

1. 目标板算力/内存有限，不适合本地完整编译。
2. 目标板系统可能精简（甚至裸机），没有完整编译环境。
3. 开发效率更高：在 PC 上编译、调试、下载到板端运行。

---

##### Build / Host / Target（面试常问）

- **Build**：构建工具运行的平台（执行 `gcc/make` 的机器）
- **Host**：编译器自身将运行的平台（多数场景与 Build 相同）
- **Target**：最终程序运行的平台

在嵌入式常见场景里可简化理解为：

`Build(PC) != Target(开发板)`

#### 描述一下嵌入式基于ROM的运行方式和基于RAM的运行方式有什么区别？

先给结论：两者核心差异在于**代码是否主要在 RAM 中运行**，这会直接影响启动流程、运行速度和可用 RAM 容量。

##### 基于 RAM 的运行方式

1. 程序启动时，通常需要先把存储介质（Flash/硬盘等）中的代码加载到 RAM，加载过程中一般会涉及重定位。
2. 运行速度通常更快（指令在 RAM 中执行），但可用 RAM 会更紧张，因为代码和数据都要占用 RAM 空间。

##### 基于 ROM 的运行方式

1. 运行速度通常相对慢一些，系统往往还需要把变量、部分代码从存储器搬移到 RAM 后再运行相应部分。
2. 可用 RAM 资源通常更多，因为并非所有代码都常驻 RAM。

---

##### 面试回答模板（30秒）

可以这样回答：

“基于 RAM 运行一般是先装载再执行，执行速度快，但 RAM 占用大；基于 ROM 运行通常 RAM 压力更小，但涉及搬移和访问存储器，整体速度往往不如纯 RAM 执行。实际项目里会按成本、性能和内存预算折中选择。”

##### 一句话记忆

**RAM 模式：快但占内存；ROM 模式：省 RAM 但速度通常略慢。**
